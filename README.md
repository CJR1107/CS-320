# CS 320 Portfolio Reflection

Software functionality can be ensured through a combination of clear requirements, structured testing, and consistent review. Unit testing is one of the most important ways to confirm that individual parts of a program work as expected. In CS 320, the contact service files demonstrated this by testing whether contacts could be created, updated, and deleted according to the required rules. Testing valid inputs is important, but testing invalid inputs is just as important because it shows whether the program can handle errors correctly. Security can also be improved by using input validation, setting limits on data, avoiding blank or invalid values, and making sure the program responds safely when unexpected information is entered.

User needs are interpreted by reviewing the project requirements and turning them into specific program features. Each requirement should become part of the software’s logic and should also be represented in the test cases. For example, if a contact field has a length limit or cannot be empty, that requirement needs to be enforced in the code and verified through testing. This process helps make sure the program is not only functional but also aligned with what the user expects. Clear requirements also reduce confusion during development because they provide a guide for both design decisions and quality assurance.

Software design should begin by breaking the overall problem into smaller, organized components. Each class or service should have a clear purpose so the program is easier to understand, test, and maintain. In Project One, separating the contact object from the contact service helped organize the program because one part represented the data and the other handled the actions performed on that data. This approach supports cleaner code and makes future updates easier. A strong software design should be simple, maintainable, and flexible enough to support changes without requiring the entire program to be rewritten.
